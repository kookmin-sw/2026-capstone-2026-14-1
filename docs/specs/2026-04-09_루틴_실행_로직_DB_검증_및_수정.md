# 2026-04-09 루틴 실행 로직 DB 검증 및 수정

## 1) 검증 기준 (docs/database_structure.md)
루틴 실행 데이터 흐름은 아래 계층을 따라야 한다.

- `routine_instance` (루틴 1회 실행)
- `routine_step_instance` (실행 단계)
- `workout_set` (단계 내 세트)
- `workout_session.set_id` (현재 수행 중 세트 연결)

목표 상태 전이:

- `routine_instance`: `RUNNING -> DONE | ABORTED`
- `routine_step_instance`: `PENDING/RUNNING -> DONE | ABORTED`
- `workout_set`: `RUNNING -> DONE | ABORTED`

## 2) 기존 문제점

- `recordWorkoutSet`가 `session_event(SET_RECORD)`만 저장하고, 실제 `workout_set / routine_step_instance / workout_session.set_id` 진행 상태를 갱신하지 않음.
- 프론트 `checkRoutineProgress`가 `REPS`만 처리하고 `TIME` 목표를 처리하지 않음.
- 세션 종료 동기화에서 루틴 상태를 무조건 `DONE`으로 덮어쓸 수 있어, 미완료 종료 케이스에서 DB 상태 왜곡 가능.
- 클라이언트 `SessionBuffer.completeSet()`에서 세트 번호 증가(`currentSetNumber++`)가 주석 라인에 묻혀 동작하지 않던 버그.

## 3) 수정 내용

### A. 백엔드 (`controllers/workout.js`)

1. 루틴 시작 컨텍스트 보강
- `createRoutineStartContext`에서 루틴 시작 시 모든 `routine_step_instance`를 생성.
- 첫 단계만 `RUNNING`, 나머지는 `PENDING`으로 시작.
- 첫 `workout_set`(`set_no=1`, `status=RUNNING`) 생성 후 `workout_session.set_id`에 연결.

2. 실행 컨텍스트 로딩 확장
- `loadRoutineExecutionBySetId` 조회 컬럼 확장:
  - `workout_set`: `set_no`, `target_type`, `target_value`, `status`, `started_at`
  - `routine_step_instance`: `step_id`, `exercise_id`, `order_no`, `planned_sets`, `completed_sets`, `status` 등

3. 세트 기록 로직 재작성 (`recordWorkoutSet`)
- `SET_RECORD` 이벤트 저장 후, 루틴 모드면 아래를 즉시 반영:
  - 현재 `workout_set`을 `DONE`으로 전환 (`actual_value`, `duration_sec`, `result_basis`, `value_unit`, `is_success` 갱신)
  - 현재 `routine_step_instance.completed_sets/status` 갱신
  - 다음 진행 분기:
    - 같은 단계 세트 남음: 다음 `workout_set` 생성, `workout_session.set_id` 이동 (`NEXT_SET`)
    - 다음 단계 존재: 다음 `routine_step_instance`를 `RUNNING`으로 전환, 첫 세트 생성, `workout_session.set_id` 이동 (`NEXT_STEP`)
    - 다음 단계 없음: `routine_instance`를 `DONE` 처리 (`ROUTINE_COMPLETE`)

4. 종료/중단 동기화 보정 (`syncRoutineExecutionFromSession`)
- `ABORTED` 동기화 시 이미 `DONE`인 루틴을 `ABORTED`로 강등하지 않음.
- `DONE` 동기화 시 현재 DB 상태를 기준으로 step/routine 완료 여부 재계산:
  - 모든 단계가 완료면 `routine_instance = DONE`
  - 미완료 종료면 `routine_instance = ABORTED`

### B. 프론트 (`public/js/workout/session-controller.js`)

1. 루틴 진행 체크 확장
- `checkRoutineProgress`를 `REPS` + `TIME` 모두 처리하도록 변경.
- 세트당 경과시간 상태(`currentSetWorkSec`)를 추가해 `TIME` 목표 달성 여부 판정.

2. 세트 완료 시 서버 동기화 추가
- 세트 완료 시 `/api/workout/session/:sessionId/set` 호출.
- 서버 저장 성공 후에만 클라이언트 세트 완료 처리(`sessionBuffer.completeSet`) 및 다음 흐름(휴식/다음 운동) 진행.
- 중복 처리 방지 락(`isCompletingSet`) 추가.

### C. 버퍼 (`public/js/workout/session-buffer.js`)

- `completeSet()`에서 누락된 `this.currentSetNumber++` 복구.
- 세트 완료 이벤트(`SET_RECORD`)는 기존처럼 기록.

## 4) 검증 결과

이번 수정으로 루틴 세션은 docs 기준 계층 구조를 따라 다음이 보장된다.

- 세트 완료 시점마다 DB의 실행 계층(`workout_set`, `routine_step_instance`, `workout_session.set_id`)이 즉시 전진.
- 루틴 종료/중단 시 최종 상태가 실제 완료 여부와 일치하도록 보정.
- `TIME` 목표 루틴도 프론트-백 동기화 경로로 정상 진행.
