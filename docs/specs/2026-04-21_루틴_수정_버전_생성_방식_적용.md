## 2. 배경
기존 루틴 수정 API는 같은 `routine_id`를 유지한 채:

1. `routine.name`을 업데이트하고
2. 기존 `routine_setup`을 전부 삭제한 뒤
3. 새 단계 구성을 다시 insert하는 방식이었다.

이 방식은 단순한 템플릿 수정처럼 보이지만, 실제 DB 구조와는 충돌 가능성이 있었다.

문서 기준 루틴 실행 계층은 아래와 같다.

```text
routine
  -> routine_setup
routine_instance
  -> routine_step_instance
    -> workout_set
      -> workout_session
```

특히 `routine_step_instance.step_id`는 실행 당시 참조한 `routine_setup.step_id`를 가리킨다.  
즉, 과거 실행 이력이 남아 있는 루틴의 `routine_setup`을 삭제하려 하면 FK 제약에 걸릴 수 있다.

## 3. 기존 문제

### 3.1 수정 시 DB 오류 발생
운동 순서, 종목, 목표값, 세트 수 같은 구성을 바꾸려고 하면 수정 API 내부에서 `routine_setup` 삭제가 먼저 발생했고, 이 과정에서 과거 실행 이력과 충돌해 오류가 반환될 수 있었다.

### 3.2 히스토리 보존 관점에서도 위험
루틴 히스토리는 `routine_instance -> routine_step_instance -> workout_set -> workout_session` 체인을 기준으로 구성된다.  
즉, 이미 실행된 루틴은 “현재 템플릿”이 아니라 “과거 실행 기록”으로 해석하는 것이 맞다.

따라서 구성 변경을 기존 템플릿에 덮어쓰는 방식은 아래 두 문제를 동시에 만든다.

- DB 참조 무결성 충돌 가능성
- 과거 기록을 참조하는 기준 템플릿을 바꾸게 되는 설계적 불일치

## 4. 변경 이유
운동 구성 수정은 사실상 “기존 루틴의 새 버전 생성”에 가깝다.  
이 프로젝트의 데이터 구조에서는 기존 루틴을 직접 갈아끼우는 것보다 새 루틴을 생성하는 방식이 더 안전하다.

이 방식을 선택한 이유는 다음과 같다.

1. 과거 실행 이력을 건드리지 않는다.
2. `routine_setup` 삭제가 필요 없어서 FK 충돌을 피할 수 있다.
3. 히스토리 탭의 과거 루틴 기록과 새 루틴 기록을 자연스럽게 분리할 수 있다.
4. 사용자 입장에서도 “이전 루틴은 남기고, 수정본을 새로 저장한다”는 개념이 더 명확하다.

## 5. 적용한 변경

### 5.1 백엔드: 수정 API를 버전 생성 방식으로 변경
파일: `controllers/routine.js`

기존 `updateRoutine`:
- 같은 `routine_id`의 이름 수정
- 기존 `routine_setup` 삭제
- 새 `routine_setup` insert

변경 후 `updateRoutine`:
- 수정 대상 루틴이 존재하는지 확인
- 해당 루틴에 `RUNNING` 상태의 `routine_instance`가 있으면 수정 차단
- 새 `routine` row 생성
- 새 `routine_id`에 대해 `routine_setup` insert
- 기존 루틴은 `is_active=false`로 비활성화
- 응답에 새 `routine_id`와 기존 `routine_id`를 함께 반환

추가 유틸:
- `countRunningRoutineInstances(userId, routineId)`

응답 예시:

```json
{
  "success": true,
  "routine_id": 123,
  "replaced_from_routine_id": 45
}
```

### 5.2 프론트: 수정 후 추가 알림 없이 목록으로 복귀
파일: `views/routine/edit.ejs`

수정 저장 성공 시:
- 별도의 성공 `alert` 없이
- 바로 `/routine`으로 이동하도록 유지했다.

이유:
- 수정 저장은 자주 일어나는 기본 동작이라 알림 노출이 오히려 흐름을 끊는다.
- 목록으로 돌아가면 새 루틴이 반영된 상태를 바로 확인할 수 있다.

## 6. 현재 최종 동작

### 6.1 새 루틴 생성
사용자가 기존 루틴 수정 화면에서 운동 구성을 바꿔 저장하면, 서버는 기존 루틴을 직접 수정하지 않고 새 루틴을 생성한다.

### 6.2 기존 루틴 비활성화
기존 루틴은 삭제되지 않고 `is_active=false`로 전환된다.  
따라서 루틴 목록에서는 사라지지만, 과거 실행 이력 보존 측면에서는 안전하다.

### 6.3 실행 중 루틴은 수정 불가
해당 루틴에 `RUNNING` 상태 실행이 있으면 새 버전 생성도 허용하지 않는다.  
실행 중 템플릿 변경으로 인한 혼선을 막기 위한 보호 장치다.

### 6.4 히스토리 보존
기존 루틴의 과거 실행 기록은 그대로 남고, 이후 새 루틴으로 실행한 기록은 새 루틴 기준으로 쌓이게 된다.

## 7. 기대 효과

1. 루틴 구성 수정 시 발생하던 DB 오류를 우회할 수 있다.
2. 과거 히스토리와 새 구성 기록이 섞이지 않는다.
3. 현재 DB 구조와 더 잘 맞는 방식으로 루틴 수명 주기를 관리할 수 있다.
4. 추후 “루틴 버전 관리” 개념으로 확장하기 쉬워진다.

## 9. 후속 고려사항

### 9.1 아카이브 표시
현재는 기존 루틴을 비활성화만 하므로 목록에서 사라진다.  
필요하면 추후 “보관된 루틴” 섹션을 별도로 둘 수 있다.

### 9.2 이름 정책
현재는 사용자가 입력한 이름으로 새 루틴이 생성된다.  
필요하면 이후 `v2`, `복사본`, `개정판` 같은 버전 표기 정책을 도입할 수 있다.

### 9.3 이름 변경만 필요한 경우
현재 수정 API는 운동 구성 수정 문제를 피하기 위해 전체를 새 루틴 생성 방식으로 처리한다.  
만약 “이름만 변경”은 기존 루틴 유지가 더 적절하다고 판단되면, 추후 이름 변경과 구성 변경을 분리한 API로 나눌 수 있다.
