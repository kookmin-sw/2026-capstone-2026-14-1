# 2026-04-09 퀘스트 DB 및 부여 로직 정리

## 1. 목적

최근 퀘스트 시스템이 **티어 기반 + 슬롯 규칙 기반 부여 구조**로 바뀌어, 실제 코드 기준 동작을 문서로 고정합니다.

- DB 기준: `docs/sql/DB_init.sql`
- 서버 로직 기준: `controllers/quest.js`, `controllers/admin.js`, `routes/main.js`

---

## 2. 변경 요약

### 2.1 핵심 변화

1. 퀘스트 템플릿에 선발 제어 컬럼 추가
- `selection_weight`
- `cooldown_days`
- `exclusive_group`
- `min_tier`, `max_tier`

2. 슬롯 규칙 테이블 도입/강화
- `quest_assignment_rule(scope, slot_no, category, type, count, tier-range)`
- 동일 `scope + slot_no + tier-range` 중복 방지 유니크 인덱스 적용

3. 부여 로직이 템플릿 단순 랜덤에서 슬롯 규칙 기반으로 변경
- 티어 필터 적용
- 쿨다운 적용
- exclusive group 충돌 회피
- 가중치 랜덤 선발

## 3. DB 구조 (퀘스트/포인트/티어)

### 3.1 `tier_rule`

- PK: `tier`
- 컬럼:
  - `tier`: 티어 숫자
  - `min_points`: 해당 티어 최소 포인트
  - `name`: 티어명

### 3.2 `quest_template`

- PK: `quest_template_id`
- 주요 컬럼:
  - `scope`: `DAILY` | `WEEKLY`
  - `type`: `DO` | `QUALITY` | `HABIT` | `CHALLENGE`
  - `category`: 슬롯 매칭용 카테고리
  - `difficulty`: `EASY` | `NORMAL` | `HARD`
  - `title`
  - `condition` (JSONB)
  - `reward_points`
  - `min_tier`, `max_tier` (nullable, `tier_rule` FK)
  - `selection_weight` (기본 100, `> 0`)
  - `cooldown_days` (기본 0, `>= 0`)
  - `exclusive_group` (nullable)
  - `is_active`, `created_at`, `updated_at`
- 제약:
  - `min_tier <= max_tier` (둘 다 있을 때)
  - `selection_weight > 0`
  - `cooldown_days >= 0`

### 3.3 `quest_assignment_rule`

- PK: `rule_id`
- 주요 컬럼:
  - `scope`: `DAILY` | `WEEKLY`
  - `slot_no`: 슬롯 번호
  - `category`, `type`: 템플릿 매칭 키
  - `count`: 슬롯에서 뽑을 개수 (`> 0`)
  - `min_tier`, `max_tier` (nullable)
  - `is_active`
- 제약/인덱스:
  - `min_tier <= max_tier` (둘 다 있을 때)
  - 유니크 인덱스:  
    `uq_quest_assignment_rule_scope_slot_tier`  
    (`scope`, `slot_no`, `coalesce(min_tier,-1)`, `coalesce(max_tier,-1)`)

### 3.4 `user_quest`

- PK: `user_quest_id`
- 주요 컬럼:
  - `user_id`
  - `quest_template_id`
  - `period_start`, `period_end`
  - `status` (`ACTIVE`, `DONE`, `EXPIRED`)
  - `progress` (JSONB)
  - `created_at`, `updated_at`
- 유니크:
  - `(user_id, quest_template_id, period_start, period_end)`

### 3.5 `point_ledger`

- PK: `ledger_id`
- 주요 컬럼:
  - `user_id`
  - `source_type` (`QUEST` 등)
  - `source_id` (퀘스트 지급은 `user_quest_id`)
  - `points`
  - `note`, `created_at`
- 중복 지급 방지:
  - `source_id IS NOT NULL`: `(user_id, source_type, source_id)` 유니크
  - `source_id IS NULL`: `(user_id, source_type)` 유니크

## 4. 퀘스트 부여 트리거

### 4.1 자동 부여 시점

- 홈 진입: `GET /`
  - `assignDailyQuests` -> `assignWeeklyQuests` 실행
- 퀘스트 페이지 진입: `GET /quest`
  - `assignDailyQuests` -> `assignWeeklyQuests` 실행

### 4.2 기간 계산

- DAILY: 오늘 00:00:00 ~ 23:59:59
- WEEKLY: 이번 주 월요일 00:00:00 ~ 일요일 23:59:59
- `user_quest.period_start/period_end`는 `YYYY-MM-DD` 날짜로 저장

## 5. 부여 알고리즘 (코드 기준)

`assignScopeQuests({ userId, scope, tier })` 기준 동작:

1. 현재 scope 기간(`period_start`, `period_end`) 계산
2. 같은 사용자/같은 기간/같은 scope에 이미 부여된 퀘스트가 있으면 종료  
   (중복 부여 방지용 idempotent)
3. 활성 규칙(`quest_assignment_rule`) + 활성 템플릿(`quest_template`) 조회
4. 사용자 티어에 맞는 규칙/템플릿만 필터링 (`min_tier`/`max_tier`)
5. 템플릿 중 최대 `cooldown_days` 산출 후, 최근 부여 이력(`user_quest`) 조회
6. 규칙(slot) 순서대로 템플릿 선발:
   - 후보: `template.type == rule.type && template.category == rule.category`
   - 같은 사이클 중 템플릿 중복 금지 (`selectedTemplateIds`)
   - 같은 사이클 중 `exclusive_group` 중복 금지
   - 쿨다운 우선 적용 후보에서 가중치 랜덤 선발
   - 쿨다운 후보가 없으면 완화 fallback으로 선발 시도
7. 선발 결과로 `user_quest` 일괄 insert
   - 초기 `progress`:
     - `kind`, `operator`, `current: 0`, `target`, `completed: false`
8. insert 시 유니크 충돌(`23505`)은 무시하여 중복 삽입 방지

### 5.1 가중치 랜덤

- `selection_weight` 합으로 roulette 방식 선발
- 가중치는 최소 1로 보정

### 5.2 쿨다운 판정

- 마지막 부여일과 현재 period start의 날짜 차이(`diffDays`) 계산
- 조건: `diffDays > cooldown_days`일 때만 쿨다운 통과

주의:
- `>` 비교이므로 예를 들어 `cooldown_days = 1`이면 **1일 차이로는 불가**, 2일 이상 차이가 나야 다시 선발됩니다.

## 6. 진행도 계산 로직

진행도는 `refreshQuestProgressForRows`에서 period 단위로 집계 후 `user_quest.progress`를 갱신합니다.

### 6.1 집계 데이터 원천

- `workout_session` (세션 수, 점수, 운동 시간)
- `routine_instance` (루틴 완료 수)

### 6.2 지원 condition kind

- `WORKOUT_SESSION_COUNT`
- `ROUTINE_COMPLETE_COUNT`
- `SESSION_SCORE_COUNT` (`min_score`, `occurrences` 사용)
- `TOTAL_SESSION_DURATION_SEC`
- `ACTIVE_DAYS_COUNT`

`status` 조건이 없으면 기본 `DONE` 기준으로 계산합니다.

---

## 7. 완료/보상 지급 로직

`claimQuestReward` 기준:

1. `user_quest` 조회
2. 이미 `DONE`이면 거절
3. 기간 만료(`period_end < today`)면 `EXPIRED`로 변경 후 거절
4. 최신 통계로 진행도 재계산
5. `completed=true` 확인 후 `user_quest.status = DONE`
6. `reward_points > 0`이면 `point_ledger`에 `source_type='QUEST'`, `source_id=user_quest_id`로 적립
7. `point_ledger` 유니크 충돌(`23505`)은 중복 지급으로 간주하고 추가 적립하지 않음

---

## 8. Admin 입력 규칙

`controllers/admin.js` 기준:

- 템플릿:
  - `scope`: DAILY/WEEKLY
  - `type`: DO/QUALITY/HABIT/CHALLENGE
  - `difficulty`: EASY/NORMAL/HARD
  - `category`: 정규식 `^[A-Z0-9_]{1,30}$`
  - `selection_weight >= 1`, `cooldown_days >= 0`
  - `condition`은 JSON 파싱 가능해야 함
  - `min_tier/max_tier`는 비우거나 1 이상 정수, `min <= max`
- 슬롯 규칙:
  - `slot_no >= 1`, `count >= 1`
  - `scope/type/category/tier-range` 검증 동일

---

## 9. 운영 체크포인트

1. 슬롯 규칙(`quest_assignment_rule`)과 템플릿(`quest_template`)의 `type/category`를 반드시 맞춰야 실제 선발됩니다.
2. 특정 티어 구간에서 규칙은 있는데 템플릿이 없으면 해당 scope 퀘스트가 생성되지 않습니다.
3. `cooldown_days`를 크게 주면 부여 실패 확률이 올라가므로, 같은 family에 템플릿 풀을 충분히 확보해야 합니다.
4. `exclusive_group`를 넓게 묶으면 같은 사이클 동시 등장 제한이 강해집니다.
5. 점수/진행도는 실시간 계산이 아니라 페이지 진입/갱신 시점에 refresh가 수행됩니다.
