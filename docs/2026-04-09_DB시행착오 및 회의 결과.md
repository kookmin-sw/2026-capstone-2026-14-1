# 2026-04-09 : 운동별 phase 정규화 DB 설계 철회 및 JS 채점 로직 전환

## 1. 문서 목적

이 문서는 **2026-04-09** 전후로 검토한 운동 판별/채점 구조 변경 사항을 정리한다.

- 기존에 검토하던 **phase 중심 DB 정규화 설계**를 왜 철회했는지
- 왜 운동별 채점 로직을 **DB 설정형이 아닌 JS 코드형**으로 전환했는지
- 최종적으로 DB에는 무엇을 저장하고, 런타임 채점은 어디서 수행하는지
- 이 결정으로 인해 admin 페이지, 프론트엔드, 백엔드의 책임이 어떻게 바뀌는지

이 문서는 단순한 결과만 적는 것이 아니라, 설계 과정에서의 **시행착오와 판단 근거**까지 함께 남기는 것을 목표로 한다.

---

## 2. 배경 및 해결하려던 문제

### 2.1 초기 목표

초기에는 운동 채점을 보다 범용적으로 관리하기 위해, 운동별 판별 로직을 **phase 단위로 분해하여 DB에 정규화**하는 방향을 검토했다.

의도는 다음과 같았다.

- 운동마다 여러 개의 phase를 정의
- 각 phase의 순서와 진입/전이 조건을 저장
- phase별 점수 산정 규칙과 피드백 규칙을 저장
- admin 페이지에서 해당 규칙들을 수정 가능하게 구성
- 동일 구조를 **일반 운동 채점**과 **운동 배우기 기능** 모두에 재사용

즉, 채점 로직을 코드에 하드코딩하지 않고 **설정 가능한 데이터 구조**로 추상화하려는 시도였다.

### 2.2 초기 설계가 기대했던 장점

- 새로운 운동 추가 시 코드 수정 없이 DB 설정만으로 대응 가능
- admin 페이지에서 phase, 전이 규칙, 피드백 규칙을 조정 가능
- 운동 배우기 기능에서도 동일한 phase 정의를 재사용 가능
- 채점 규칙이 테이블 단위로 관리되어 일관성 있는 운영 가능

### 2.3 실제 검토 과정에서 드러난 문제

팀 회의와 구조 검토 결과, 이 방식은 이론적으로는 깔끔하지만 실제 서비스 운영에서는 다음과 같은 문제가 컸다.

- 운동마다 phase 구조가 일정하지 않다.
- 일부 운동은 phase 기반으로 표현 가능하지만, 일부는 phase보다 더 복잡한 조건 분기나 예외 처리가 필요하다.
- 운동별로 필요한 metric, 판별 기준, 보정 방식, 예외 처리 방식이 크게 다르다.
- admin 페이지에서 운동별로 phase와 전이 규칙, 피드백 규칙을 모두 관리하게 하면 설정 오버헤드가 매우 커진다.
- DB 정규화 구조가 복잡해질수록 실제 채점 로직의 유연성이 오히려 떨어진다.
- 결국 복잡한 운동은 JSON condition이나 예외 필드를 계속 늘리게 되어, “설정형 구조”가 아니라 “우회적으로 코드를 DB에 넣는 구조”가 될 가능성이 높았다.

핵심적으로, **운동별 판별 방식이 동일한 추상화로 수렴하지 않는다**는 점이 가장 큰 문제였다.

## 3. 아키텍처 결정

### 3.1 phase 중심 DB 정규화 설계 철회

최종적으로, 운동 판별/채점 로직을 phase 중심으로 일반화하여 DB에 정규화하는 방향은 철회했다.

철회 이유는 다음과 같다.

- 운동별 로직 편차가 커서 범용 phase 모델로 수용하기 어렵다.
- admin에서 세부 규칙까지 관리하도록 만드는 비용이 크다.
- 구조를 일반화할수록 실제 구현은 더 복잡해지고 디버깅도 어려워진다.
- 향후 새로운 운동이 늘어날수록 DB 스키마보다 코드 복잡도가 문제의 본질이 된다.

### 3.2 운동별 채점 로직은 JS 코드로 분리

최종 결정은 **각 운동별 채점 로직을 JS 코드로 직접 구현**하는 것이다.

즉,

- 운동마다 별도 JS 채점 모듈을 둔다.
- 해당 운동에 필요한 판별 방식은 그 운동 코드 내부에서 자유롭게 구현한다.
- phase 기반 운동이면 phase 방식으로 구현할 수 있다.
- phase가 아닌 다른 구조가 더 적합하면 그 방식 그대로 구현할 수 있다.
- 공통 엔진이 모든 운동을 강제로 같은 방식으로 처리하지 않는다.

이 방식은 범용성은 줄지만, 실제 운동 로직의 다양성과 복잡성을 그대로 수용할 수 있다.

### 3.3 DB의 책임 축소

채점 로직 자체는 코드로 이동시키고, DB는 **운영에 필요한 최소한의 메타데이터와 결과 저장소** 역할만 담당하도록 책임을 축소한다.

DB는 다음 정보만 관리한다.

- 운동 목록
- 운동별 허용 자세(view) 규칙
- 세션/실행 이력
- 채점 결과 요약
- 세부 metric 결과 및 최종 점수

즉, DB는 “채점 규칙 엔진”이 아니라 **채점 대상 메타정보와 결과 히스토리 저장소**가 된다.

## 4. 최종 데이터 흐름

### 4.1 운동 선택 및 자세 선택

사용자는 운동을 선택하고, 해당 운동에서 허용된 자세 값 중 하나를 선택한다.

허용 자세 값은 다음 3가지로 단순화한다.

- `FRONT` : 정면
- `SIDE` : 측면
- `DIAGONAL` : 대각선

운동마다 이 3개 중 허용 가능한 값만 DB에 저장한다.

예시:

- 스쿼트: `FRONT`, `SIDE`
- 런지: `SIDE`, `DIAGONAL`
- 특정 운동: `FRONT`만 허용

### 4.2 프론트엔드/런타임 채점

운동과 자세 값이 결정되면, 해당 운동에 연결된 JS 채점 로직이 실행된다.

이 로직은 다음을 자유롭게 수행할 수 있다.

- landmark 기반 판별
- phase 기반 상태 전이
- 특정 metric 계산
- rep 카운팅
- 예외 자세 처리
- 자세별 다른 기준 적용
- 최종 점수 계산
- 실시간 피드백 생성

중요한 점은, 이 로직이 더 이상 DB에 정의된 범용 phase 규칙을 해석하는 방식이 아니라, **운동별 맞춤 코드가 직접 판단**한다는 것이다.

### 4.3 백엔드 저장

채점이 끝나면 프론트엔드 또는 채점 모듈은 다음 결과를 백엔드로 전달한다.

- 운동 ID
- 선택한 자세 값
- 세션 정보
- 반복 횟수 또는 수행 횟수
- 세부 metric 결과
- 최종 점수
- 요약 피드백
- 필요 시 추가 detail payload

백엔드는 이를 관련 결과 테이블에 저장한다.

즉, 백엔드는 채점 규칙을 다시 계산하지 않고, **채점 결과를 기록하고 조회 가능한 형태로 저장**하는 역할에 집중한다.

## 5. 최종 설계 상세

### 5.1 DB에 남기는 정보

#### `exercise`

- 운동 기본 정보
- 운동 코드, 이름, 설명, 활성화 여부 등

#### `exercise_allowed_view` 또는 동등한 구조

- 운동별 허용 자세 목록 저장
- 값은 `FRONT`, `SIDE`, `DIAGONAL` 중 하나
- 한 운동은 여러 자세를 허용 가능

이 부분은 배열 컬럼으로 관리할 수도 있고, 별도 테이블로 정규화할 수도 있다.
다만 현재 목적상 중요한 것은 **운동마다 허용 자세를 명시할 수 있어야 한다**는 점이다.

#### `workout_session` /  결과 테이블

- 사용자가 어떤 운동을 어떤 자세로 수행했는지
- 어떤 metric들이 계산되었는지
- 최종 점수가 얼마였는지
- 어떤 피드백이 생성되었는지
- 추후 기록 조회 및 통계에 필요한 detail

### 5.2 코드에 두는 정보

운동별 JS 코드에는 다음 내용이 들어간다.

- 사용할 metric 정의
- landmark 해석 방식
- phase 방식 여부
- rep 인식 방식
- threshold 및 보정 로직
- 자세(view)에 따른 분기
- 실시간 피드백 규칙
- 최종 점수 계산식

즉, 기존에 DB로 옮기려 했던 핵심 채점 규칙은 이제 **운동별 JS 모듈의 책임**이 된다.

### 5.3 admin 페이지 역할 변경

기존 구상에서는 admin 페이지에서 다음을 모두 관리하려 했다.

- 운동별 phase 정의
- phase 전이 규칙
- 피드백 규칙
- 점수 산정 규칙

하지만 최종 구조에서는 admin 페이지의 역할을 축소한다.

admin 페이지에서 관리할 대상은 주로 다음과 같다.

- 운동 등록/수정
- 허용 자세 설정
- 운동 활성화 여부
- 필요 시 설명/메타데이터 수정

즉, admin은 더 이상 “채점 규칙 편집기”가 아니라 **운영용 운동 메타데이터 관리 페이지**가 된다.

## 6. 이 결정으로 얻는 이점

### 6.1 운동별 복잡성을 자연스럽게 수용 가능

운동마다 판별 방식이 달라도 억지로 하나의 DB 모델에 끼워 맞출 필요가 없다.

- phase 기반 운동은 phase 방식으로 구현 가능
- 비phase 기반 운동은 별도 흐름으로 구현 가능
- 예외 처리나 특수 로직도 직접 코드에서 처리 가능

### 6.2 admin 오버헤드 감소

운동별로 수많은 규칙을 운영자가 직접 설정하지 않아도 된다.

- 설정 실수 가능성 감소
- 복잡한 관리 UI 구현 부담 감소
- 실제 서비스 운영 난이도 감소

### 6.3 디버깅과 유지보수 단순화

문제가 발생했을 때 DB의 여러 규칙 테이블을 추적하는 대신, 해당 운동의 JS 로직만 보면 된다.

- 원인 파악이 빠르다
- 테스트 단위가 명확하다
- 운동별 수정 범위를 국소화할 수 있다

### 6.4 결과 저장 구조는 유지 가능

채점 로직을 코드로 옮기더라도, 결과는 여전히 구조화해서 저장할 수 있다.

따라서 다음 기능은 계속 유지 가능하다.

- 운동 기록 조회
- metric별 결과 분석
- 점수 히스토리 누적
- 사용자 피드백 이력 확인
- 향후 통계/추천 기능 확장

## 7. 한계 및 트레이드오프

### 7.1 설정형 범용성은 줄어든다

새 운동을 추가할 때 DB 설정만으로 끝나지 않고, 운동별 JS 로직 작성이 필요하다.

즉, “완전한 no-code 설정형 채점 시스템”은 포기한 셈이다.

### 7.2 코드 자산 관리가 중요해진다

운동 수가 늘어날수록 JS 채점 모듈이 많아진다.
따라서 다음이 중요해진다.

- 운동별 모듈 구조 통일
- 공통 유틸 분리
- metric 계산 함수 재사용
- 테스트 코드 또는 시뮬레이션 도구 마련

### 7.3 일부 공통 규칙은 별도 추상화가 필요할 수 있다

모든 운동이 완전히 독립적이면 중복 코드가 늘 수 있다.
따라서 추후에는 DB 정규화가 아니라 **코드 레벨 공통 엔진/유틸** 형태로 재사용성을 확보하는 것이 더 적절하다.

## 8. 최종 결론

이번 검토를 통해, 운동 판별/채점 로직을 phase 중심으로 일반화하여 DB에 정규화하는 방식은 실제 요구사항 대비 과도한 추상화라는 결론에 도달했다.

최종적으로는 다음 구조를 채택한다.

- DB는 운동 목록과 허용 자세 규칙을 저장한다.
- 사용자는 운동과 허용된 자세 값을 선택한다.
- 실제 채점은 해당 운동의 JS 코드에서 수행한다.
- 채점에 사용된 세부 metric과 최종 점수는 백엔드로 전달한다.
- 백엔드는 이를 결과 테이블에 저장한다.

즉, **운영 메타데이터는 DB에 두고, 운동별 판별 복잡성은 코드가 담당하며, 결과는 다시 DB에 저장하는 구조**로 정리한다.

이 방식은 범용 설정형 구조보다 덜 일반적이지만, 실제 운동 로직의 다양성과 구현 현실을 더 정확하게 반영하는 방향이다.

---

## 9. 향후 구현 방향

- `exercise` 기준으로 운동별 JS 채점 모듈 매핑
- 운동별 허용 자세 값 저장 구조 확정
- 프론트엔드에서 운동 + 자세 선택 후 해당 로직 호출
- 채점 결과 payload 형식 표준화
- 백엔드 히스토리 저장 스키마 정리
- 공통 metric 유틸 및 점수 계산 보조 함수 분리
