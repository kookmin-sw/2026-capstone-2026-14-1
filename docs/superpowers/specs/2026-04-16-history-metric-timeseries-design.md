# 히스토리 상세 메트릭 시계열 그래프 설계

## 1. 문서 정보

- 작성일: 2026-04-16
- 대상 프로젝트: FitPlus
- 대상 화면: `views/history/index.ejs`의 세션 상세 모달
- 목적: 세션 진행 중 메트릭 점수 변화(`avg_score`)를 시간축 기준으로 시각화하는 1차 설계를 고정한다.

## 2. 배경

현재 시스템은 운동 종료 시점에 아래 데이터를 이미 저장한다.

- `workout_session`
- `session_snapshot`
- `session_snapshot_score`
- `session_snapshot_metric`
- `session_event`

그러나 히스토리 상세 API는 현재 최종 스냅샷의 메트릭만 읽어 화면에 보여준다. 이 때문에 사용자는 최종 점수는 볼 수 있어도, 세션 중 어떤 메트릭이 언제 좋아지거나 나빠졌는지는 확인할 수 없다.

## 3. 목표

- 히스토리 상세 모달에서 메트릭별 점수 변화를 시간축으로 보여준다.
- 첫 버전은 `avg_score`만 사용해 운동 종류와 무관하게 같은 스케일(`0~100`)로 읽히게 한다.
- 기존 저장 구조를 바꾸지 않고, 이미 저장된 스냅샷/메트릭 데이터를 재구성해서 사용한다.

## 4. 비목표

- LLM 분석 추가
- `avg_raw_value` 중심 그래프 제공
- 여러 메트릭을 한 그래프에 동시에 겹쳐 그리는 비교 모드
- 히스토리 목록 페이지 전체 개편
- 운동 결과 페이지(`views/workout/result.ejs`)까지 동일 시각화 확장

## 5. 권장 접근과 대안

### 권장안: 상세 API 확장 + 프론트 단일 메트릭 라인 차트

- `controllers/history.js`의 상세 API가 `INTERIM`과 `FINAL` 스냅샷 메트릭을 모두 조회한다.
- 서버에서 메트릭별 시계열 구조를 만들고, 프론트는 이 데이터를 받아 단일 메트릭 라인 차트를 그린다.

장점:

- 현재 구조와 가장 잘 맞는다.
- 변경 범위가 `history` 상세 조회와 모달 UI에 집중된다.
- 프론트는 렌더링 책임만 가져서 코드 경계가 비교적 선명하다.

단점:

- 상세 API 응답 크기가 약간 늘어난다.

### 대안 1: 서버가 차트용 데이터를 더 많이 가공

- 서버가 차트 좌표와 정렬까지 모두 완성해서 내려준다.

장점:

- 프론트 단순화

단점:

- 표현 변경이 생기면 백엔드 변경 폭이 커진다.

### 대안 2: 스냅샷 raw 데이터를 그대로 내려주고 프론트에서 전부 재구성

장점:

- API 자체는 단순하다.

단점:

- `public/js/history-page.js`가 더 비대해진다.
- 데이터 정렬/병합 책임이 프론트로 과도하게 이동한다.

최종 선택은 권장안으로 한다.

## 6. 데이터 설계

### 6.1 조회 대상

- `session_snapshot`
  - `session_snapshot_id`
  - `snapshot_no`
  - `snapshot_type`
  - `recorded_at`
- `session_snapshot_metric`
  - `session_snapshot_id`
  - `metric_key`
  - `metric_name`
  - `avg_score`
  - `avg_raw_value`
  - `min_raw_value`
  - `max_raw_value`
  - `sample_count`

### 6.2 시간축 기준

첫 버전은 다음 두 값을 모두 유지한다.

- `snapshot_no`: 정렬과 디버깅용 순서값
- `t_sec`: `recorded_at - started_at` 기준 상대 시간(초)

차트 X축은 `t_sec`를 우선 사용한다.

### 6.3 응답 구조

기존 상세 응답에 `metric_series`를 추가한다.

```json
{
  "metric_series": [
    {
      "metric_key": "depth",
      "metric_name": "스쿼트 깊이",
      "points": [
        {
          "snapshot_no": 1,
          "snapshot_type": "INTERIM",
          "recorded_at": "2026-04-16T10:00:05.000Z",
          "t_sec": 5,
          "avg_score": 42,
          "sample_count": 14
        },
        {
          "snapshot_no": 2,
          "snapshot_type": "INTERIM",
          "recorded_at": "2026-04-16T10:00:12.000Z",
          "t_sec": 12,
          "avg_score": 55,
          "sample_count": 18
        },
        {
          "snapshot_no": 3,
          "snapshot_type": "FINAL",
          "recorded_at": "2026-04-16T10:00:21.000Z",
          "t_sec": 21,
          "avg_score": 61,
          "sample_count": 24
        }
      ]
    }
  ]
}
```

## 7. 서버 설계

### 7.1 상세 API 확장

`controllers/history.js`의 상세 조회 흐름에서 아래를 추가한다.

- `INTERIM`과 `FINAL` 스냅샷을 모두 조회한다.
- 해당 스냅샷 ID에 연결된 `session_snapshot_metric`를 모두 조회한다.
- 메트릭별로 그룹화하고 `snapshot_no` 기준으로 정렬한다.
- 각 포인트에 `t_sec`를 계산해 넣는다.

### 7.2 서버 책임

서버는 아래까지만 책임진다.

- 조회
- 정렬
- 메트릭별 그룹화
- 시간축 계산

차트 SVG/HTML 구성 책임은 프론트에 둔다.

## 8. 프론트 설계

### 8.1 UI 구성

세션 상세 모달에 아래 블록을 추가한다.

- 메트릭 선택 영역
- 선택 메트릭 요약 영역
- 라인 차트 영역

### 8.2 첫 버전 동작

- 메트릭이 여러 개면 첫 번째 메트릭을 기본 선택한다.
- 한 번에 하나의 메트릭만 라인 차트로 보여준다.
- 요약 카드에는 최소한 다음을 보여준다.
  - 메트릭 이름
  - 시작 점수
  - 마지막 점수
  - 변화량
  - 샘플 수 또는 포인트 수

### 8.3 차트 표현

- Y축은 `0~100`
- X축은 상대 시간(`t_sec`)
- 점과 선을 같이 표시한다.
- 포인트가 1개뿐이면 선 대신 점 중심 UI로 처리한다.

## 9. 예외 처리

- 중간 스냅샷 메트릭이 없고 최종 메트릭만 있는 세션:
  - 차트 대신 "시계열 데이터가 충분하지 않아 최종 메트릭만 표시합니다." 안내를 보여준다.
- 메트릭명이 비어 있으면 `metric_key`를 대신 표시한다.
- 메트릭 포인트가 정렬되지 않았더라도 서버에서 정렬해 일관성을 보장한다.
- 점수가 비정상 값이면 기존 점수 처리 규칙에 맞춰 `0~100` 범위로 제한한다.

## 10. 테스트 및 검증

### 10.1 서버 검증

- 스냅샷이 여러 개인 세션에서 메트릭 시계열이 올바르게 묶이는지 확인
- `INTERIM`이 없는 세션에서 예외 없이 응답하는지 확인
- 서로 다른 운동 코드에서도 응답 구조가 동일한지 확인

### 10.2 프론트 검증

- 메트릭 선택 전환 시 차트가 정상 갱신되는지 확인
- 포인트 1개 세션, 포인트 여러 개 세션 모두 렌더링 확인
- 모바일 폭에서도 모달 내부 차트가 깨지지 않는지 확인

## 11. 구현 순서

1. `controllers/history.js` 상세 API에서 전체 스냅샷 메트릭 조회 추가
2. 서버에서 `metric_series` 구조 생성
3. 상세 응답에 `metric_series` 포함
4. `views/history/index.ejs`에 그래프 섹션 마크업 추가
5. `public/js/history-page.js`에 메트릭 선택/차트 렌더 로직 추가
6. 실제 세션 데이터로 동작 확인
7. 테스트 실행 및 최종 검증

## 12. 결정 사항 요약

- 지금 단계의 우선순위는 LLM이 아니라 기존 시계열 데이터 시각화다.
- 첫 버전은 `avg_score` 기반 메트릭 시계열만 다룬다.
- 한 그래프에는 한 메트릭만 표시한다.
- 구현 범위는 히스토리 상세 모달로 한정한다.
